参考答案

（1）技术处理细节如下：
（A）浏览器处理输入：自动在网址上添加默认的协议和端口号‌，并在本地查找是否有这个网址的缓存版本；
（B）DNS 解析：查询域名对应的 IP 地址，依次在本地 Host 文件、本地域名服务器、根域名服务器中查找；
（C）建立连接：通过 TCP 三次握手与服务器建立连接；
（D）发送 HTTP 请求：若使用 HTTPS 协议，还会进行 SSL 握手，建立加密信道‌，然后设置请求头发出请求；
（E）服务器处理请求：根据请求解析出受访的主机、应用、资源，封装并返回 HTTP 响应报文；
（F）浏览器解析响应和渲染页面：如果响应头中的 Content-Type 是 text/html，则进行 HTML 解析，否则做其他处理‌；在渲染页面时，如果遇到外部资源链接（如图片、JS 文件等），则会进一步请求这些资源。

（2）可使用以下 JVM 参数进行调优：
（A）‌堆内存设置：

  • -Xms：设置初始堆空间。由于服务器内存只有 1G，建议设置为较小值，如 -Xms256m 或 -Xms512m，确保程序启动时不会占用过多内存‌；另外，由于资源受限，建议将初始堆空间（-Xms）和最大堆空间（-Xmx）设置为相同的值，以避免堆内存的动态扩展和收缩带来的性能开销。
  • -Xmx：设置最大堆空间。同样地，考虑到内存受限和性能开销，应设置为一个合适的值，且最好与 -Xms 设置的值相同，如 -Xmx256m 或 -Xmx512m，也能避免内存溢出，同时留出一定空间供操作系统和其他程序使用‌，确保服务器的稳定。

（B）垃圾回收器选择：

  • -XX:+UseG1GC：启用 G1 垃圾回收器。G1 回收器能够更灵活地管理内存，减少垃圾回收带来的停顿，提高程序的响应速度‌。
  • -XX:+UseSerialGC：使用串行垃圾回收器。在单核 CPU 和有限内存的环境中，串行垃圾回收器是一个合理的选择，因为它不需要额外的线程来处理垃圾回收，从而减少了 CPU 和内存的消耗。

（C）线程栈大小设置：

  • ‌-Xss：设置每个线程的栈大小。在内存有限的情况下，将其设置为较小值，如 -Xss256k 或 -Xss512k，以节省内存并允许创建更多的线程‌，提高线程创建和管理的效率。但也要注意不能设置得过小，如果方法调用的深度过大，可能会导致栈溢出错误。

（D）性能监控参数：

  • ‌-XX:+PrintGCDetails‌：输出详细的 GC 日志。这有助于了解垃圾回收的行为和性能瓶颈，从而进行针对性的优化。
  • -XX:+PrintGCDateStamps：在 GC 日志中添加时间戳。可以更准确地分析 GC 事件的时间分布。
  • -Xloggc:：将 GC 日志输出到指定文件。便于收集和分析 GC 数据，如 -Xloggc:log/gc.log。

（3）NIO 主要解决了在高并发、高性能需求下，使用传统阻塞 I/O 模型所带来的资源消耗大、效率低下等问题。传统的阻塞 I/O 一个线程管理一个 socket，而 NIO 只用一个线程或少数几个线程就能管理多个网络连接，效率大大提高，特别适用于需要同时处理大量并发连接或大量数据传输的场景，比如网络服务器、文件传输系统等。它的原理是基于事件驱动的 I/O 模型，使用多路复用器（Selector）和事件通知机制来实现非阻塞 I/O 操作。核心概念包括：

  • Selector：多路复用器，可以注册多个 Channel，并监听这些通道上的特定事件，比如读取数据、写入数据等；
  • Channel：通道，数据传输的载体，比如文件通道、网络连接通道等，它是双向的，既可以读取也可以写入；
  • Buffer：缓冲区，一个固定大小的内存块，用于存储要读取或写入的数据，所有数据都通过它来处理。

具体工作流程如下：

  • 创建一个 Selector，并将其绑定到某个线程上；
  • 创建通道，并将其注册到 Selector 上，指定监听的事件类型；
  • 当通道上的事件到达时，Selector 就将该事件通知给应用程序；
  • 应用程序通过调用 Selector 的方法获取已经到达的事件列表；
  • 根据不同的事件类型，应用程序进行相应的读取、写入或其他操作。

以 Netty 为例，它的调优技巧有：
（A）线程池：根据应用负载和硬件配置，设置合适的线程池大小，如核心线程数、最大线程数；BossGroup 通常设置为单线程，负责处理连接请求；WorkerGroup 负责处理 I/O 操作，可以设置为 CPU 核心数 × 2；
（B）内存：启用 Netty 的内存池功能，通过循环利用 ByteBuf 对象来减少内存分配和回收的开销；
（C）协议：根据具体的应用场景选择合适的协议（如 UDP、WebSocket、TCP 等），并使用更加高效的序列化框架，比如选择 Protobuf 来替代 JSON；
（D）I/O 模型：根据应用场景和操作系统选择合适的 I/O 模型，比如在 Linux 系统上用 epoll 会更加高效；
（E）编解码器：选择合适的编解码器，对于大数据量传输，使用 Netty 自带的工具处理粘包和拆包问题，如 LengthFieldBasedFrameDecoder 和 DelimiterBasedFrameDecoder；
（F）日志：调整日志级别来减少不必要的日志输出，提升性能；
（G）网络配置：开启 TCP_NODELAY 选项，减少延迟，提高小数据包的传输效率；调整 SO_BACKLOG 选项，增加等待连接队列的长度；
（H）使用压缩：对于大数据量的 I/O，可以考虑使用压缩技术（如 Gzip）来减少数据传输量，提高 I/O 性能。

（4）方案如下：
（A）使用外部存储（如 Redis）集中存储 Session 数据，所有服务器统一对外部存储读取或写入；
（B）使用文件来存储 Session 数据，不同服务器之间通过共享这些文件来操作 Session 数据；
（C）搭建一个 Session 服务器专门管理 Session 数据，其他服务器统一访问这个 Session 服务器。

Session 是一种在服务器端用于跟踪用户会话信息的机制，它的出现主要是为了解决 HTTP 协议无状态的问题，以及为用户提供更好的服务体验。无状态即每个请求都是独立的，服务器不会记住与客户端之前的交互信息，但在实际应用中，用户通常希望在多个请求之间将一些状态信息保持一段时间，如登录状态、购物车内容等。它的用途常有：
（A）用户认证‌：通过 Session 可以判断用户是否已经登录，并获取用户的身份信息；
（B）状态保持‌：可用于在多个页面请求之间保持用户的状态信息，如购物车内容、表单填写进度等；
（C）个性化服务‌：根据 Session 中的用户信息，服务器可以提供个性化服务，如内容推荐、偏好设置等。

在 Java 开发中，Session、Cookie 和 Token 都是用于管理用户会话和身份验证的重要机制，以下是三者的区别：
Session：存储在服务器端，相对安全，适合需要长时间保持会话状态的应用（当用户首次登录后，服务器会创建并保存一个唯一的 Session，并将 Session ID 通过 Set-Cookie 或其他方式发送给客户端，客户端在后续的请求中会携带这个 Session ID，服务器通过验证 Session ID 来识别用户身份和会话状态）；
Cookie：存储在客户端 / 浏览器中，安全性较低，可设置 HttpOnly 增加安全性，适合需要频繁访问的应用（服务器在 HTTP 响应中设置 Cookie，比如就可以将 Session ID 或者 Token 设置到 Cookie 中，客户端浏览器在后续的请求中会自动携带这些 Cookie，自动登录功能的核心就是要先判断 Cookie 是否存在，然后再判断 Cookie 里面保存的用户信息是否正确，正确即可登录）；
Token：通常存储在客户端，更加安全，因为可以设置过期时间，适合需要跨域访问或增强安全性的应用（拥有无状态的特性，完美适配 HTTP，当用户首次登录成功后，服务器会生成一个包含用户信息和权限的 Token，Token 的生成通常包含选择加密算法（比如 RSA 算法）、签名（需要密钥）、编码（比如 Base64 编码）几个步骤，生成成功后将其返回给客户端，客户端在后续的请求中会携带这个 Token，作为请求头的一部分，服务器通过验证 Token 就能识别用户身份，避免频繁查询数据库，减轻服务器的压力）。
针对不同的应用场景，这三者可以相互配合同时使用，比如在一个应用里，一些功能需要保持会话状态，另一些功能又需要请求是无状态的，那就可以同时用 Session 和 Token，利用各自的优势，满足多样化的需求。

（5）线程间通信指的是在多线程编程中，不同线程之间如何交换信息或协调执行。常见的实现方式有：
（A）共享内存：多个线程可访问同一块内存，通过读写这块内存实现数据交换，如全局变量，但要注意同步问题；
（B）消息传递：线程之间通过发送和接收消息来进行通信，比如使用消息队列；
（C）条件变量‌：通过条件变量实现线程间的等待和唤醒，适用于线程需要等待某个条件成立才能继续执行的场景；
（D）信号量：这是一种用于线程同步和互斥的机制，它维护一个计数器，表示可用资源的数量，线程通过等待（P 操作）和释放（V 操作）信号量来实现同步，适用于需要控制对共享资源的访问的场景；
（E）管道：数据只在一个方向上流动（半双工），比如生产者-消费者模型；
（F）事件：线程需要等待某个事件（如 I/O 操作完成）才能继续执行。

在 Java 中的实现：
（A）volatile：保证共享变量的可见性和有序性（配合 synchronized 或 ReentrantLock 使用，保证原子性）；
（B）BlockingQueue：提供了线程安全的队列实现，可用于在生产者和消费者线程之间传递数据；
（C）wait 和 notify/notifyAll：由 Object 类提供，用于线程间的等待和通知，必须与 synchronized 配合使用；
（D）Condition：提供了比 wait / notify 更为灵活的通信机制，通常与 ReentrantLock 配合使用；
（E）Semaphore：在 java.util.concurrent 包中，通过 acquire 方法获取许可，通过 release 方法释放许可；
（F）CountDownLatch：允许一个或多个线程等待其他线程完成操作（老师必须要等到所有学生交完试卷才能走）；
（G）CyclicBarrier：允许一组线程互相等待，直到所有线程都到达某个屏障点（比如，多人游戏必须都要加载到 100% 才能开始）；
（H）Exchanger：一个用于在线程之间交换数据的工具。

并发编程的最佳实践：
（A）减少共享变量的使用：通过设计，将对共享变量的访问降到最低，尽量使用不可变对象和局部变量；
（B）使用线程安全的集合类：比如 ConcurrentHashMap、CopyOnWriteArrayList 等，这些类是线程安全的；
（C）选择合适的同步工具：比如 synchronized、ReentrantLock 或 java.util.concurrent 包中的工具类。
（D）确保所有线程都以相同的顺序获取锁，避免死锁；
（E）使用线程池。

synchronized 和 Lock 的区别：
（A）synchronized：是一个内置的关键字，可用于方法和代码块，不需要显式地获取和释放锁，当线程进入一个 synchronized 方法或代码块时，会自动获得锁，退出时自动释放锁，只支持抢占式的非公平锁，采用悲观锁机制，适用于简单的同步场景；
（B）Lock：是一个接口，常用的实现类有 ReentrantLock，需要显式地调用 lock() 方法获取锁，通常在 finally 块中调用 unlock() 方法确保锁的释放，还提供了 tryLock()、isLocked() 等方法实现更灵活的控制，支持公平锁和非公平锁，可在创建对象时指定，采用乐观锁机制，适用于复杂的同步场景。

（6）超时处理的优化策略如下：
（A）提高任务执行效率：通过优化算法、减少不必要的 I/O 操作等方式，缩短执行时间；
（B）合理分配资源：确保系统有足够的资源来处理任务（如内存、网络带宽等）；
（C）使用缓存：对于频繁访问但不经常变化的数据，可使用缓存来减少时间；
（D）异步处理：使用异步编程来处理任务（如 CompletableFuture），减少等待时间；
（E）使用更高效的超时检测机制：比如使用 ScheduledThreadPoolExecutor 定期检查状态，取代阻塞式的 get；
（F）快速失败：如果可以检测到任务不可能在超时时间内完成，就尽快取消任务并释放资源；
（G）友好的错误提示：发生超时时，向用户显示清晰的错误提示和可能的解决方案；
（H）重试机制‌：提供重试按钮或自动重试功能，让用户可以方便地重新尝试操作；
（I）进度条或加载指示器：在操作进行时显示进度条或加载指示器，让用户知道并非无响应。

（7）适合存储的数据类型包括：
（A）String：适合存储任何形式的字符串，比如软件版本号、API 的执行结果、用户使用的语种（SET / GET）；
（B）Hash：适合存储具有多个属性的对象信息，比如数据字典、用户信息、商品信息（HSET / HGETALL）；
（C）List：适合存储需要有一定顺序的数据集合，比如任务列表、操作日志、消息队列（LPUSH / RPOP）；
（D）Set：适合存储不能有重复的数据集合，如在线用户、标签、类别、抽奖参与人员（SADD / SMEMBERS）；
（E）SortedSet：适合存储需要根据分数进行排序的数据集合，比如排行榜、时间线（ZADD / ZRANGE，OTA 时，用收到终端心跳时的当前时间作为分数，每收到一次心跳就更新分数，这样就能根据分数删除心跳超时的终端 ID，以此获取当前在线的可更新的终端列表）。

击穿指的是缓存中某个热点数据突然失效（因过期或被淘汰），导致大量请求直接访问数据库，造成数据库压力骤增‌。常见于秒杀商品、热点新闻等场景，解决方法：
（A）使用一个互斥锁（比如 Redis 的 SETNX 或 Redisson 分布式锁），仅允许一个线程访问数据库并重建缓存，其他线程等待锁释放后读取新缓存；
（B）不设置数据的物理过期时间，而是在数据中添加一个逻辑过期字段，当发现数据过期时，再异步更新缓存；
（C）在热点数据快过期前主动异步更新，降低击穿概率‌。

雪崩指的是缓存中大量数据同时失效，或 Redis 宕机，导致所有请求直接访问数据库，可能引发数据库崩溃。常见于大促期间缓存集中过期等场景，解决方法：
（A）对缓存数据的过期时间添加随机值，避免批量数据同时失效‌；
（B）增加二级缓存（如 Memcached），当 Redis 缓存失效时，可从二级缓存读取；
（C）部署 Redis 主从集群或哨兵模式，结合数据库限流策略，防止单点故障导致服务不可用‌；
（D）对关键数据开启持久化功能，配合 Hystrix 等熔断工具，在数据库压力过大时暂时拒绝部分请求‌。

  • RDB 持久化：生成某一时刻内存数据的二进制快照文件（默认名为 dump.rdb），可通过 SAVE（阻塞主线程） 或 BGSAVE（后台异步子线程） 命令手动生成，也可通过配置让其自动生成。
  • AOF 持久化：以日志的形式记录 Redis 服务器处理过的每一个写操作命令（默认名为 appendonly.aof，且默认的 appendfsync 同步频率为每秒同步一次），然后在 Redis 服务器重启时通过执行该文件中的命令来恢复数据‌。可与 RDB 混合使用，结合 RDB 的快速恢复和 AOF 的低数据丢失风险，兼顾性能与安全性。
  • 哨兵模式：负责监控主从节点的健康状况（发送 PING 命令检测），并在主节点故障时自动触发故障转移（需多个哨兵节点投票确认主节点是否下线，通常哨兵节点为奇数个），将合适的从节点提升为新的主节点（多个哨兵节点选举产生），同时通过发布订阅模式通知客户端更新主节点地址‌，确保服务持续可用。

穿透指的是请求的数据在缓存和数据库中均不存在，导致请求持续绕过缓存直接访问数据库‌。常见于恶意或异常请求等场景（比如攻击者故意查询不存在的数据），解决方法：
（A）对请求参数进行合法性校验，拦截明显无效的请求‌；
（B）使用布隆过滤器预加载合法数据（如有效用户 ID），请求时先通过过滤器判断数据是否存在，若不存在，则直接返回（建议选择可扩展的布隆过滤器应对数据动态增长‌，并定期重建过滤器，避免误判率随时间累积升高‌）；
（C）当查询结果为「数据不存在」时，将空值（如 NULL 或特定标记字符串）写入缓存并设置较短过期时间（如 5 - 30 分钟），后续相同请求直接返回缓存中的空值，需注意监控空值缓存的内存占用，防止恶意攻击导致内存溢出‌。

（8）可从以下几个方面进行说明：
（A）掌握 Linux 的基本命令，比如 mkdir、pwd、chmod、cp、mv、history、crontab、vi 编辑器的使用等等；
（B）能熟练使用 Linux 的系统管理命令，比如 useradd、passwd、ps、top、kill、systemctl、df、apt、yum 等；
（C）能熟练使用 Linux 的命令行工具，比如 tar、ping、ifconfig、netstat、grep、awk 等等；
（D）能编写 Shell 脚本实现自动化任务，常用语法包括：使用 #!/bin/bash 指定脚本解释器，使用 $ 符号引用变量值，使用 -eq、-lt、-gt 来比较数字大小，使用 echo 来输出字符串或覆盖文件内容等等；
（E）了解 Linux 的内存管理，它是 Linux 内核的一个关键部分，负责内存的分配与回收，使用了虚拟内存技术。

（9）RPC（Remote Procedure Call，远程过程调用）是一种用于实现分布式系统中不同节点之间通信和调用的技术。实现一个 RPC 框架通常包括以下步骤：
（A）设计‌协议：解决服务端与客户端通信的问题，包括网络传输协议、序列化协议和消息编码协议；
（B）实现‌服务端：实现代理对象，接收和处理客户端的请求，调用对应的本地方法，并将结果返回给客户端；
（C）实现‌客户端：实现代理对象，发起 RPC 调用，将调用参数打包成请求，并发送给远程服务器；
（D）服务注册与发现：将服务提供者的地址和接口等信息注册到注册中心，消费者能通过注册中心发现服务；
（E）负载均衡与容错机制：实现负载均衡策略来分配请求，以及容错机制来保证调用的稳定性。

一次完整的 RPC 调用流程如下：

  • 服务消费者（Client）以本地调用方式调用服务；
  • Client stub 将调用的参数等信息封装为消息体，然后找到远程服务地址将消息发送；
  • Server stub 收到消息后进行解码，根据解码结果调用服务提供者（Server）的方法；
  • Server stub 将服务提供者的方法执行结果打包成消息发送给 Client stub；
  • Client stub 收到返回消息后进行解码，将最终结果返回给服务消费者。

RPC 框架的目标就是将上面 Client stub 和 Server stub 要做的事情都封装起来，让用户感知不到它们的存在，使用时就像在调用本地方法一样简单，而不用关心网络通信的细节。RPC 主要用到的技术有动态代理、序列化、NIO 等，常见的 RPC 框架有 Dubbo、gRPC（基于 HTTP/2 实现，支持多路复用；支持多种编程语言，通过 .proto 文件定义的接口可以自动生成代码；使用 ProtoBuf 进行序列化，相比与 HTTP 常用的 JSON 文本来说传输效率更高）、Hessian（基于 Binary-RPC 协议，走二进制数据） 等，支持 RPC 功能的框架有 WebService、Spring Cloud 等。

（10）SOA 为一种面向服务的架构（Service-Oriented Architecture），是一种软件设计和开发的方法，它将应用程序拆分成一组独立的服务，这些服务之间通过定义良好的接口进行通信。简单来说，可以把 SOA 想象成一套积木，每个积木都是一个独立的服务，都有自己的功能，它们可以组合在一起形成更大的应用程序。

SOA 与微服务的区别：SOA 注重复用性和统一的架构，倾向于通过统一的服务总线（ESB）将所有服务连接起来，使用统一的数据存储，服务之间有一定的依赖关系，通常需要一起部署，相比于微服务来说更加复杂。微服务更加强调服务的自治，每个服务都有自己独立的环境、数据存储和技术栈等，倾向于创建更加小型、轻量级的服务，服务之间没有依赖关系，支持独立部署和持续集成，相比于 SOA 来说更加轻量。

（11）缓存技术是提高系统性能和响应速度的关键组件，在高并发的场景下尤为重要。它的作用有：
（A）减轻数据库压力：通过缓存频繁访问的数据，减少查询数据库的次数，可避免数据库成为性能瓶颈；
（B）提高数据读取速度：缓存通常存储在内存中，而内存的访问速度远快于磁盘；
（C）降低网络延迟：对于需要从远程服务器获取的数据，缓存可以减少对远程服务器的请求次数。

缓存技术的核心原理是将频繁访问的数据存储在读写速度更快的存储介质中（如内存），以减少对慢速存储介质的访问次数（如磁盘或数据库），那么哪些数据会被频繁访问呢？基于数据的局部性原理，最近被访问过的数据很可能在不久的将来再次被访问，并且与它相邻的数据也很可能被访问。当用户请求数据时，系统会首先检查缓存中是否存在所需的数据，如果存在，则直接返回缓存中的数据，从而大大缩短数据访问时间，如果不存在，则查询并添加至缓存。常见的缓存类型有本地缓存（Spring Cache）、分布式缓存（Redis、Memcached）、CDN 缓存（内容分发网络，是一种通过‌分布式节点服务器‌将图片、视频、网页等内容缓存到全球多个地理位置，使用户能够从离自己最近的节点快速获取资源的技术）、数据库缓存（EHCache、OSCache）等，常见的缓存失效策略有 LRU、LFU、‌FIFO 等。

缓存技术的缺点：
（A）额外的硬件支出：缓存需要额外的磁盘空间和内存空间来存储数据，增加了硬件成本；
（B）数据一致性问题：缓存与数据库之间可能存在数据不一致的问题，需要采取额外的措施来保证数据一致性；
（C）高并发缓存失效问题‌：在高并发场景下，缓存失效可能会导致数据库访问量瞬间增大，甚至导致数据库崩溃。

缓存的数据一致性问题主要由以下原因引起：
（B1）‌数据更新与缓存同步不一致：更新数据源后，缓存中的数据未能及时更新（Cache-Aside、Write-Through）；
（B2）‌并发更新：多个并发的更新操作导致缓存中的数据更新顺序与数据源中的更新顺序不一致（使用分布式锁）；
（B3）缓存失效策略不当：缓存中的数据已过期，但由于没有有效的失效策略而仍然存在（设置合理的失效策略）。

（12）过程如下：
（A）请求首先到达 Servlet 容器（如 Tomcat，已被内置在 Spring Boot 中）；
（B）然后依次经过过滤器链（Filter Chain），每个过滤器可以对请求进行预处理操作，并决定是否将请求继续传递（即执行 chain.doFilter() 方法之前的代码，如对请求进行认证、校验等）；
（C）到达前端控制器 DispatcherServlet，它负责分发请求，根据请求的 URL 和请求方式找到对应的控制器；
（D）依次经过拦截器链（Interceptor Chain），每个拦截器可以对请求进行预处理操作，并决定是否将请求继续传递（即执行 preHandle() 方法中的代码，如对请求进行校验、参数绑定等，但这步是可选的，因为拦截器目前只在 Spring MVC 中有效，而 Spring Boot 集成并自动配置了 Spring MVC）；
（E）执行 Controller 中的具体方法，并返回响应数据；
（F）响应数据经过拦截器链，执行 postHandle() 方法中的代码，进行后处理；
（G）执行异常解析器来处理异常（如全局异常处理），然后渲染视图；
（H）执行拦截器链中的 afterCompletion() 方法，进行关闭资源等善后工作；
（I）响应数据经过过滤器链，执行 chain.doFilter() 方法之后的代码。

（13）措施如下：
（A）定位问题：
（a）启用慢查询日志：通过设置阈值来记录执行时间过长的 SQL，以快速定位高频慢查询语句‌。

（b）使用 EXPLAIN 关键字分析 SQL 语句的执行计划，确保使用到了索引，重点关注：

  • type 字段：访问类型（如全表扫描 ALL、索引扫描 INDEX 等）；
  • key 字段：实际使用的索引；
  • rows 字段：预估扫描行数‌。

（B）索引优化：
（a）为高频查询条件（如 WHERE、JOIN、ORDER BY 后的字段）创建索引，并定期检查。

（b）避免索引失效：

  • 避免隐式类型转换（如对字符串类型的字段使用数字去查询）；
  • 避免索引列在 WHERE 子句中参与运算（如 WHERE YEAR(create_time) = 2025）；
  • 避免使用左模糊匹配（如 LIKE ‘%keyword’），如必须使用，可以考虑用全文索引代替；
  • 避免使用 OR、NOT IN 和 NOT EXISTS 引发全表扫描‌‌。

（C）调整 SQL 结构：

  • 避免使用 SELECT *，明确指定所需字段以减少 I/O 开销‌；
  • 用 ORDER BY 和 LIMIT 等子句减少查询结果的数量；
  • 将多层子查询拆解为临时表或内存计算‌；
  • 用 JOIN 替代 IN 或 EXISTS 子查询，降低执行复杂度‌。

（D）配置调优：

  • 增大 innodb_buffer_pool_size 缓存池，提升热数据访问速度‌；
  • 优化 max_connections 和连接池配置，避免资源争用‌；
  • 根据业务场景选择合适的事务隔离级别。

（E）架构级优化：

  • 读写分离，建立集群，并通过主从分离将读请求分发到从库，降低主库压力‌；
  • 分库分表，对超大数据表进行水平拆分（如按时间或 ID 范围分），分散单表压力‌。

（F）工具辅助：

  • 使用压测工具 sysbench 模拟高并发场景，验证优化效果；
  • 集成 Prometheus + Grafana 监控慢查询、锁等待等关键指标‌。

（14）防止被提前读取的方案有：
（A）目标文件上传完成后（服务端是不知道何时传完的），再上传一个跟它文件名相同，但后缀不同的临时文件，通过这个临时文件来判断目标文件的上传情况。比如要上传的目标文件是 users.xlsx，上传完成后另外再上传一个文件 users.tmp，这样一来，服务端如果想要读文件 users.xlsx，就要先判断文件 users.tmp 是否存在，若存在，则说明文件 users.xlsx 已经上传完成，可以放心去读了。
（B）改目标文件名：比如要上传的目标文件是 data.txt，那么在文件上传完成前，我们可以用脚本将文件名改为类似 data.txt.uploading 这样的格式，等文件上传完成后，再将其改回原来的 data.txt 即可，外部程序仅监控正式文件 data.txt，避免读取未完成文件‌。
（C）将上传中的文件存放在临时目录（如 /temp），并设置严格的访问权限（如仅上传进程可读写），待文件上传完成后，再将其移动至正式存储目录‌，确保服务端的文件仅在完整上传后才可见‌。

进一步了解下客户端与服务端通信时，防止消息被中途抓包的方法：
（A）强制使用 HTTPS / TLS 协议，加密传输数据，防止明文被拦截；并对敏感数据（如账号、交易信息）使用 AES、RSA 等算法进行二次加密，这样即使 HTTPS 被破解也仍然无法解密业务数据；
（B）增加数字签名，服务端校验签名与数据完整性，防止篡改‌；
（C）增加数字证书，客户端与服务端互相验证证书，避免单向认证导致中间人攻击；
（D）自定义私有的二进制协议并增加校验位，替代 JSON / XML 等明文协议；
（E）将数据包拆分并乱序传输，接收端按预设规则重组，降低抓包工具解析成功率‌；
（F）关键算法（如 AES 密钥派生）通过 JNI 调用 C++ 实现，增加逆向分析难度；
（G）通过自动化工具模拟抓包攻击，检测协议漏洞并及时修复，必要时更新加密算法和证书‌。
